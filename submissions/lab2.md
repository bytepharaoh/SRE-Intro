### 2.7 — How does the gateway find the events service?

The gateway calls `http://events:8081/...` using the hostname `events`. When this DNS name is resolved, Docker's embedded resolver at `127.0.0.11` maps it to `172.18.0.5` — the events container's IP on `app_default`. This is Docker Compose's automatic service discovery: every service name in `docker-compose.yaml` becomes a resolvable hostname for all other containers on the same network. No manual IP configuration is needed, and if a container restarts and gets a new IP, the DNS record updates automatically.

## Task 2 — Dockerfile Optimization

### Image sizes before and after `.dockerignore`

**Before:**
```
app-events:latest     3c149836f9f8        260MB         57.6MB
app-gateway:latest    3816871a047a        239MB         52.3MB
app-payments:latest   a42db8fc515c        236MB         51.8MB
```

**After (`docker compose build --no-cache`):**
```

app-gateway:latest    5c2a5ad78a81        239MB         52.3MB
app-payments:latest   1dc97b6b3d6f        236MB         51.8MB
```

The sizes are identical. This is expected — the build context for each service is just `main.py` and `requirements.txt`, so there was nothing to exclude in the first place. The `.dockerignore` would make a difference if there were a `.git/` directory, test files, or large local artifacts inside the service folder being sent to the daemon unnecessarily.


### Non-root user — Dockerfile change

Added to all three Dockerfiles (`app/gateway/Dockerfile`, `app/events/Dockerfile`, `app/payments/Dockerfile`) before the `CMD` line:

```dockerfile
RUN addgroup --system app && adduser --system --ingroup app app
USER app
```

Since the Dockerfiles are new files tracked by git (not modified from a previous commit), `git diff` shows nothing — the full content is visible via `git show` or `git status`. The relevant addition to each Dockerfile is the two lines above.

**Verification — whoami after rebuild:**

```
$ docker compose up -d --build
$ docker exec app-gateway-1 whoami
app
```

The container now runs as the unprivileged `app` user instead of `root`. This matters because if the application process is compromised, the attacker gets a system user with no shell and no privileges rather than full root access to the container filesystem.

## Bonus Task — Request Trace Across All 3 Services

**Setup:** fresh `docker compose down && docker compose up -d`, waited 5 seconds, then ran a full reserve → pay flow for reservation `c900f1e3-4a8f-4a4c-962d-23590dd227ea`.

**Full annotated trace:**

```

            → gateway received the /events/1/reserve request and forwarded it to events

[T+0ms]
events-1   | 2026-06-12T10:59:08.540326Z  Reserved 1 tickets for event 1: c900f1e3...
            → events decremented the ticket count in postgres and wrote the hold to redis

[T+~1ms]
events-1   | 2026-06-12T10:59:08.540932Z  POST /events/1/reserve HTTP/1.1  200 OK
            → events returned the reservation_id to gateway

[T+~1ms]
gateway-1  | 2026-06-12T10:59:08.542284Z  POST /events/1/reserve HTTP/1.1  200 OK
            → gateway returned reservation response to the client (curl)

--- client immediately calls /reserve/c900f1e3.../pay ---

            → payments responded to gateway

[T+~225ms]
gateway-1  | 2026-06-12T10:59:08.767238Z  POST http://events:8081/reservations/c900f1e3.../confirm  (200 OK)
            → gateway called events again to confirm the order with the payment reference

[T+~225ms]
events-1   | 2026-06-12T10:59:08.766440Z  Order confirmed: c900f1e3...
            → events updated the reservation status to confirmed in postgres

[T+~226ms]
events-1   | 2026-06-12T10:59:08.766864Z  POST /reservations/c900f1e3.../confirm HTTP/1.1  200 OK
            → events responded to gateway

[T+~227ms]
gateway-1  | 2026-06-12T10:59:08.767971Z  POST /reserve/c900f1e3.../pay HTTP/1.1  200 OK
            → gateway returned the final confirmed order to the client
```

**Total end-to-end time for the pay call: ~227ms**
(from `10:59:08.541` gateway receiving /reserve to `10:59:08.767` returning the pay response)

**Hop breakdown:**
- gateway → events (reserve): ~1ms
- client → gateway → payments (charge): ~220ms  
- gateway → events (confirm): ~4ms

The payments call dominates at ~220ms, which makes sense — it's a mock that simulates processing time. The internal hops between gateway and events are sub-millisecond, showing that Docker's bridge network adds virtually no latency for container-to-container calls on the same host.
